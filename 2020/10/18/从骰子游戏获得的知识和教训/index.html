关于HashMap的教训

1.hashcode()和equals() 这两个自带的判别函数并不是完全可靠的，如果这两哥们不能发挥作用，最好自己写一个

2.对于HashMap，如果没有特别的操作，当录入两个Key值一样而Value值不一样的时，后录入的Value值会覆盖前一个录入的Value值，如果可以，最好避开这种情况，或者搞点骚操作来处理这种情况

3.HashMap的toSpring()就是个大骗子！如果你存入HashMap的类型不是自定义的类，那么调用toSpring()会直接调用Object类的toSprig()，重写都救不回来（当然，也可能是我操作不到位）

~~ 当然如果你是自己定义的类，记得重写toSpring()函数，可以有效提高观感体验（欧洲程序员的审美，呸！）~~

    HashMap.Entry<Integer,Integer[]> entry = null; //用来遍历HashMap的所有元素

    for (Mt.entrySet().iterator() ; iterator.hasNext() ;){
        entry = (HashMap.Entry) iterator.next();
        //这不是强化for，这就是个普通的for循环
        //这个里面随便你写啥
        }

这个是通过迭代器Iterator和Entry对HashMap进行遍历，当然还有其他方法，我不熟

5.对HashMap操作时，若将KV对定义为包装类，那么这个HashMap就只能操作包装类。

关于Integer和int

1.Integer是int型的包装类，是对象；int是基本类型，这两个玩意儿不一样

关于Integer和int的具体区别，看看这个：int 和 Integer的区别

2.当需要对Integer和int经行转换时，不能和强制类型转换一样搞。（真的，绝壁不止我一个这么干过）

3.一般而言，对于单个元素进行比较时，int在需要时会自动装箱转化为Integer；而Integer在需要时会自动拆箱为int（其他时候该咋的还是咋的，需要的时候还是要自己来）

4.关于单个元素进行int和Integer间的转化

//int 到 Integer
int a = 3;
Integer A = new Integer(a);

//Integer 到 int
Integer A = new Integer(3);
int a = A.intValue();
//另外，Integer.parselnt(String str)是将String转化为int型的函数

5.关于int数组 和 Interger数组进行转换（依旧使用的是Integer的相关函数）

int[] a = {1,2,3};
Interger[] B = {1,2,3};

//int[] 转 Integer[]
Integer A = IntStream.of(a).boxed().toArray(Integer[] :: new)
//这个式子还有其他的写法，比如
Integer A = IntStream.of(a).boxed().collect(Collectors.toList()).toArray(new Integer[0]);
//这种写法也可以将int数组改造为List链表，去掉toArray即可
List<Integer> ls = IntStream.of(a).boxed().collect(Collectors.toList());

//Integer[] 转 int[]

关于时间的格式化输出

其实这次的题目在时间类上所受到的挫折较少，并没有什么特别需要提出来的地方。

1.关于时间类的定义，用Calendar来专门获取一个对象是没有必要的，如果单纯是想获得当前时间，可以通过Galendar的函数直接获取，经行拼接即可

2.在经行格式化输出时，规定的格式和所输入的格式必须完全一致，不然会抛出错误，一定要完全一致

3.TimeFormat可以将Date的输出模式替换成我所需要的输出模式

关于文件的读写

本次题目通过 FileOutputStream,OutputSreamWriter,BufferWriter 三个类完成了对答案的读写，如图

解题时IDEA的控制台画面：

读写的文件画面

1.每次文件在读写时，会重新打开文件进行读写，若多次执行读写操作，就会造成覆盖，文件中只会残留最后一次操作的数据

2.在读写结束后需要关闭流时，只需要关闭最外层的BufferWriter即可，但是必须关闭操作流

3.在读写的内容上，\n,\t等转义字符不会被写入文件中，和printf的效果是相同的
